ALINX AXU2CGB
Dual-Core Ping-Pong

Two Cortex-A53 cores running simultaneously on the Zynq UltraScale+ MPSoC, exchanging messages through shared DDR memory — no RTOS, no BSP, no HAL. A foundation project for understanding multi-core PS programming and the XSCT debug workflow on the ALINX AXU2CGB developer board.

Project Overview

How It Works

The board boots PetaLinux from SD card, which initializes the DDR4 memory controller. Once Linux is fully up, XSCT loads two bare-metal ELF files onto individual A53 cores via JTAG — bypassing FSBL entirely. Core 0 writes "Ping!" to UART and sets a shared flag; Core 1 waits on that flag, writes "Pong!", and clears it. The cycle repeats indefinitely.

A single volatile unsigned int at a fixed DDR address acts as a mutex-free turn indicator between the two cores. The volatile keyword prevents the compiler from caching the flag in a register — without it, each core would loop forever on a stale value.

System Architecture

Block Diagram

Linux boots on all four A53 cores normally. After boot, XSCT halts Core 0 and Core 1, loads the respective ELF files, and releases both. The cores run bare-metal code while Linux continues running on Core 2 and Core 3 — and has already initialized DDR, making it available for bare-metal use.

Why boot Linux first? Vitis 2025's FSBL — compiled with GCC 13 + LTO — produces DWARF debug info that XSDB cannot parse. The XFsbl_Exit breakpoint always times out, preventing ELF download. Booting PetaLinux instead initializes DDR, then XSCT loads bare-metal ELFs directly onto individual cores — a clean workaround requiring zero FSBL involvement.
Implementation Notes

Key Concepts

🔄

Shared Memory Flag

A single volatile uint32_t at 0x50000000 acts as the turn indicator. Core 0 waits for flag == 0, writes Ping, sets flag = 1. Core 1 waits for flag == 1, writes Pong, sets flag = 0.

No mutex · No atomics · volatile is enough here
📍

Separate Load Addresses

Each ELF is linked to a different DDR region via its linker script. Core 0 starts at 0x10000000, Core 1 at 0x20000000. Without this separation, both ELFs overwrite each other in memory.

lscript.ld · ORIGIN changed per core
🖥

Bare UART Output

No printf, no BSP, no stdlib. Each character is written directly to the UART TX FIFO register after polling the status register to confirm the FIFO is not full.

0xFF00002C status · 0xFF000030 TX
⏱

Software Delay

A volatile loop counter prevents the compiler from optimizing away the delay. At 1.2 GHz, 300 million iterations give roughly 1–2 seconds between messages.

300_000_000 iterations ≈ ~1–2 s
⚡

XSCT Workflow

A single TCL script connects to the hardware, resets and loads each core independently, then releases both. One command replaces the entire Vitis Run flow.

xsct run.tcl · connect → dow → con
🔧

Vitis 2025 Migration

Migrating from Vitis 2020 revealed that GCC 13 LTO corrupts DWARF sections, making XSDB unable to find XFsbl_Exit. The Linux DDR-init workaround fully sidesteps the issue.

GCC 13 · -flto · DWARF5 incompatibility

core0_ping.c — Cortex-A53 #0
void delay(void) {
    volatile int i;
    for(i = 0; i < 300000000; i++);
}

int main() {
    volatile unsigned int *tx     = (volatile unsigned int*)0xFF000030;
    volatile unsigned int *status = (volatile unsigned int*)0xFF00002C;
    volatile unsigned int *flag   = (volatile unsigned int*)0x50000000;

    char *ping = "Ping!\r\n";
    char *p;

    *flag = 0;  /* Core 0 goes first */

    while(1) {
        while(*flag != 0);  /* Wait for Core 1 to clear the flag */

        p = ping;
        while(*p) {
            while(*status & 0x10);  /* Wait for TX FIFO space */
            *tx = *p++;
        }

        delay();
        *flag = 1;  /* Signal Core 1 */
    }
    return 0;
}

run.tcl — XSCT automation script
# ZynqMP Dual-Core Ping-Pong — XSCT run script
# Usage: xsct run.tcl
# Prerequisite: PetaLinux must be fully booted before running.

connect

# Load Core 0
targets -set -filter {name =~ "Cortex-A53 #0"}
rst -processor -clear-registers
dow /path/to/core0_ping/build/core0_ping.elf

# Load Core 1
targets -set -filter {name =~ "Cortex-A53 #1"}
rst -processor -clear-registers
dow /path/to/core1_pong/build/core1_pong.elf

# Release both cores
targets -set -filter {name =~ "Cortex-A53 #0"}
con
targets -set -filter {name =~ "Cortex-A53 #1"}
con
